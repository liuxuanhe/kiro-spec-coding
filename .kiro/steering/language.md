# 对话与输出语言规范（Kiro 必须严格遵守）

- 全程对话语言（Chat）
  - 从本次会话开始，到 Spec Coding 结束为止，你在与用户的所有自然语言交流 **所有回复中的自然语言部分必须使用中文（简体）**，包括中间推理、解释、进度说明、错误提示等。
  - 禁止在同一条回复中切换为英文长句；除非用户在该条消息中明确要求使用英文，否则你不得使用英文句子来解释需求、设计或任务。
  - 你可以使用英文仅限于：
    - 代码块、配置、API Path、JSON 字段、类名/方法名/错误码/环境变量等技术标识。
    - EARS 关键字（`WHEN`/`SYSTEM`/`SHALL` 等）和本 Spec 明确要求保留英文的专有名词。
- 代码注释使用中文。
- 代码中的变量名、类名、方法名等标识符保持英文。


- 若用户未在该条消息中明确要求英文：
  - 你不得因为是“后续第二句/第三句”就切换成英文说明；
  - 若你已经输出了英文解释，需要立即用中文补充等价说明，并在后续回复中坚持使用中文。

- 验收口径：
  - 如果在 task 执行的 chat 对话中，除代码/关键字部分外仍大量出现英文长句解释，视为不符合本项目语言规范，必须重新用中文改写后再继续。

# 文档与代码语言规范

- 所有说明文档（`requirements.md` / `design.md` / `tasks.md`）的叙述性内容一律使用中文（简体）。
- 所有代码相关元素统一使用英文，包括但不限于：
  - 类名、方法名、变量名
  - API Path（如 `/api/v1/owners/register`）
  - HTTP Header（如 `Authorization`、`X-Request-Id`）
  - JSON 字段名（如 `communityId`、`houseNo`）
  - 错误码（如 `PARKING_1001`）
  - 环境变量（如 `SPRING_DATASOURCE_URL`）
  - 数据库表名和字段名（如 `sys_community`、`community_id`）
